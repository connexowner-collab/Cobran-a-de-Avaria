Critérios de Aceite e Regras de Negócio

Projeto: Gestão de Usuários – Portal do Cliente

Versão: 2.0 (detalhado com rastreabilidade código)

Objetivo: Documento de referência para a equipe de Operações e desenvolvimento, com critérios de aceite e regras de negócio redigidos em estilo PO, explicados em mínimo detalhe e vinculados ao código-fonte por IDs únicos.

Como obter .docx: Abra este arquivo no Word (Arquivo > Abrir > este .html) e use Arquivo > Salvar como > Documento do Word (*.docx).

1. Convenção de IDs e Rastreabilidade

Tipos de ID:

  • CA-Uxx – Critério de aceite (Usuários).
  • CA-Gxx – Critério de aceite (Grupos de cliente).
  • CA-Pxx – Critério de aceite (Perfis de acesso).
  • CA-Axx – Critério de aceite (Ativos e placas).
  • RN-Uxx, RN-Gxx, RN-Pxx, RN-Axx – Regras de negócio (Usuários, Grupos, Perfis, Ativos).

Uso: Cada ID aparece no documento com explicação detalhada e, em seguida, na seção Rastreabilidade com o(s) arquivo(s) e o trecho de código (função, componente, rota API) que implementa ou valida aquele item. A Matriz de rastreabilidade (Seção 2) concentra ID → arquivo → referência no código para consulta rápida.

3. Detalhamento por Área

3.1 Usuários

Como operador do portal, quero cadastrar e editar usuários com nome, e-mail, vínculo a grupo(s)/divisão(ões)/cliente e perfil de acesso, para que cada pessoa acesse apenas o que for autorizado e eu possa auditar e ajustar acessos sem depender de suporte técnico.

CA-U01 – Validação de nome e e-mail obrigatórios

Explicação detalhada: O sistema não pode permitir que um usuário seja criado ou atualizado sem nome ou sem e-mail, pois esses campos são essenciais para identificação e para login no portal. A validação ocorre em duas camadas: (1) na interface, antes do envio do formulário, bloqueando o submit e exibindo mensagem clara; (2) na API, garantindo que mesmo chamadas diretas (ex.: integração) não persistam dados inválidos. O texto exibido ao usuário deve ser único e objetivo: "Preencha nome e e-mail."

Rastreabilidade CA-U01: Front-end: src/app/gestao-usuarios/novo/page.tsx (handleSubmit, ~L74-76), src/app/gestao-usuarios/[id]/page.tsx (handleSubmit, ~L129-131) — setError('Preencha nome e e-mail.'). API: src/app/api/usuarios/route.ts POST (~L47-51), src/app/api/usuarios/[id]/route.ts PUT — if (!nome || !email) return NextResponse.json({ error: 'Campos obrigatórios: nome e email' }, { status: 400 }).

CA-U02 – Exigência de ao menos um vínculo (grupo, divisão ou cliente)

Explicação detalhada: Todo usuário deve estar vinculado a pelo menos um contexto de acesso: um ou mais grupos de cliente (grupoIds ou o legado grupoId), uma ou mais divisões (divisaoIds) ou, em último caso, um cliente (clienteId). Sem isso, o usuário não teria escopo de dados no portal. A regra é aplicada na tela (impedindo envio e exibindo "Adicione ao menos um grupo ou divisão de cliente.") e na API (retornando 400 com a mensagem longa que inclui os nomes dos campos técnicos para integração).

Rastreabilidade CA-U02: Front-end: novo/page.tsx (~L78-80), [id]/page.tsx (~L133-135). API: src/app/api/usuarios/route.ts POST (~L53-59): ids.length === 0 && divIds.length === 0 && !clienteId → 400, error: 'Informe ao menos um grupo de cliente (grupoIds ou grupoId), divisão (divisaoIds) ou cliente (clienteId)'.

CA-U03 – Perfil de acesso obrigatório na criação

Explicação detalhada: Na criação do usuário, a PO define que é obrigatório selecionar um perfil de acesso. O perfil carrega o conjunto padrão de permissões (módulos e funcionalidades) e pode ser ajustado depois na edição; mas na criação, não permitimos usuário sem perfil para evitar acessos indefinidos. A mensagem na interface é: "Selecione um perfil de acesso." A validação é feita apenas no front (novo/page.tsx), pois na edição o perfil pode ficar em branco se o usuário foi criado antes da regra ou por migração.

Rastreabilidade CA-U03: src/app/gestao-usuarios/novo/page.tsx handleSubmit (~L82-85): if (!perfilId) { setError('Selecione um perfil de acesso.'); return; }. Select de perfil: estado perfilId, opção vazia "Selecione um perfil".

CA-U04 – Unicidade de e-mail (rejeição com 409)

Explicação detalhada: Não podem existir dois usuários com o mesmo e-mail (considerando normalização: trim e conversão para minúsculas). A verificação é feita na API em POST (criação) e em PUT (edição); na edição, o próprio usuário sendo editado é excluído da checagem (excluirUsuarioId). Em caso de conflito, a API retorna HTTP 409 Conflict com a mensagem "Já existe um usuário cadastrado com este e-mail." para que a interface ou o integrador exibam feedback claro.

Rastreabilidade CA-U04: src/app/api/usuarios/route.ts: função emailJaCadastrado(email, excluirUsuarioId); POST ~L61-66: if (emailJaCadastrado(emailNorm)) return NextResponse.json({ error: 'Já existe...' }, { status: 409 }). src/app/api/usuarios/[id]/route.ts PUT ~L43-48: mesma função com id como excluirUsuarioId.

CA-U05 – Perfil persistido e exibido na edição

Explicação detalhada: O perfil escolhido no cadastro (ou na última edição) deve ser salvo no objeto do usuário (campo perfilId) e persistido no store. Ao reabrir o usuário para edição, o dropdown de perfil deve mostrar o perfil salvo e a matriz de permissões deve refletir as permissões do usuário (que podem ter sido herdadas do perfil e eventualmente customizadas). Para não sobrescrever as permissões já salvas na primeira carga da tela de edição, o código usa um useRef (skipNextPerfilApply) para ignorar o primeiro efeito que aplica as permissões do perfil ao trocar perfilId.

Rastreabilidade CA-U05: src/app/gestao-usuarios/[id]/page.tsx: carregamento do usuário ~L58-61 (setPerfilId(usuario.perfilId ?? ''), setPermissoes(usuario.permissoes), skipNextPerfilApply.current = true). src/lib/store.ts: criarUsuario e atualizarUsuario incluem perfilId. src/app/api/usuarios/[id]/route.ts PUT: perfilId no body e no objeto update.

CA-U06 – Matriz de permissões somente leitura fora de "Criar perfil"

Explicação detalhada: As alterações de módulos e funcionalidades (quais telas e ações o usuário pode acessar) devem ser feitas exclusivamente na tela "Criar perfil", para centralizar a governança. Nas telas de criação e edição de usuário, a matriz é apenas visualização: os checkboxes estão desabilitados, a área tem pointer-events-none e opacidade reduzida, e o texto explicativo orienta o usuário a usar "Criar perfil" para alterar. Isso evita que operadores alterem permissões de forma fragmentada por usuário.

Rastreabilidade CA-U06: src/components/MatrizPermissoes.tsx: prop readOnly?: boolean; em toggle e toggleModulo if (readOnly) return; container com readOnly ? 'pointer-events-none select-none opacity-90'; labels e inputs disabled={readOnly}. src/app/gestao-usuarios/novo/page.tsx e [id]/page.tsx: <MatrizPermissoes ... readOnly />. ModalCriarPerfil usa MatrizPermissoes sem readOnly.

CA-U07 a CA-U09 – Listagem, filtros e exclusão

CA-U07: A coluna "Perfil de acesso" aparece na tabela de usuários antes da coluna "Status". O valor exibido é o nome do perfil (resolvido a partir de perfis carregados via GET /api/perfis); se o usuário não tiver perfilId, exibe "—". Código: src/app/gestao-usuarios/page.tsx — estado perfis, loadPerfis(), getPerfilNome(u), <th> e <td> da coluna, FilterInput para filtroColPerfil.

CA-U08: O operador pode filtrar a lista por nome, e-mail, grupo de cliente, perfil de acesso, status e por intervalo de datas (data de criação e último acesso). Cada coluna tem um campo de filtro; as datas usam DateRangeFilter. Código: page.tsx — usuariosFiltered com matchCol e isDateInRange; estados filtroCol*, filtroCriadoEmRange, filtroUltimoAcessoRange.

CA-U09: Ao excluir um usuário, se o ID for inválido ou o usuário não for encontrado (ex.: já excluído), a API retorna 400 ou 404 com mensagem adequada; a interface exibe o erro via setMensagemFlash. Código: page.tsx handleExcluir; src/app/api/usuarios/[id]/route.ts DELETE: validação de id, getUsuarioById, 404 'Usuário não encontrado', 400 para ID inválido.

3.3 Perfis de Acesso

Como operador, quero criar perfis com nome e matriz de módulos/funcionalidades, para que eu reutilize o mesmo conjunto de permissões em vários usuários e altere regras em um único lugar.

CA-P01: O nome do perfil é obrigatório. Na API, retorno 400 "Nome do perfil é obrigatório" se nome for vazio ou só espaços; no modal Criar perfil, setError('Informe o nome do perfil.'). src/app/api/perfis/route.ts POST ~L21-26; src/components/ModalCriarPerfil.tsx ~L33.

CA-P02: Perfis são persistidos no arquivo .data/store.json (junto com usuarios e grupos). Ao fechar e reabrir a aplicação, os perfis continuam disponíveis. saveToFile é chamado após criarPerfil; loadFromFile recarrega perfis na inicialização e reloadStoreFromFile() é usado nas rotas da API. src/lib/store.ts: STORE_FILE, saveToFile, loadFromFile, criarPerfil → saveToFile. src/app/api/perfis/route.ts: reloadStoreFromFile() no GET e POST.

CA-P03: Na tela "Criar perfil" a matriz de permissões é editável (MatrizPermissoes sem readOnly). Nas telas de usuário (novo e editar), a matriz é somente leitura (readOnly). ModalCriarPerfil não passa readOnly para MatrizPermissoes; novo/page e [id]/page passam readOnly. MatrizPermissoes.tsx implementa desabilitar toggles e estilo quando readOnly.

3.4 Ativos e Placas

Como operador, quero que ativos com ou sem placa sejam suportados e exibidos sem traço na placa, para que equipamentos identificados só por número de série também possam ser vinculados a grupos e divisões.

CA-A01: Na listagem de ativos, a coluna de identificação exibe placa quando existir; caso contrário, número de série (a.placa || a.numeroSerie). O tipo Ativo tem placa opcional e numeroSerie obrigatório. src/types/index.ts Ativo: placa?: string; numeroSerie: string. ModalNovoGrupo.tsx e store: exibição (a.placa || a.numeroSerie). Store: ativos a129–a133 sem placa.

CA-A02: Placas são armazenadas e exibidas sem hífen (ex.: ABC1D23). Não há formatação que insira traço no código. Store seed e dados: placa como 'ABC1D23', etc. Tipo string; nenhum replace ou formatação com hífen em exibição ou persistência.

CA-A03: Ativos sem placa (apenas numeroSerie) participam normalmente da lista de ativos do grupo, podem ser selecionados por checkbox e movimentados para divisões. A movimentação usa ativoIds; não há restrição por presença de placa. getAtivosByContratoIds retorna todos os ativos dos contratos; ModalNovoGrupo lista todos; moverAtivosEmLoteParaDivisao recebe ativoIds (incluindo de ativos sem placa).

4. Regras de Negócio (Resumo e Referência ao Código)

As regras RN-Uxx, RN-Gxx, RN-Pxx e RN-Axx estão descritas na Matriz de rastreabilidade (Seção 2) com o arquivo e a referência no código. Abaixo, o resumo de cada regra com ênfase no "o que" e "por quê"; o "onde" está na matriz.

  • RN-U01: E-mail normalizado (trim e minúsculas) para unicidade e consistência — API usuarios.
  • RN-U02: Múltiplos grupos (grupoIds) e/ou divisões (divisaoIds); grupoIds tem prioridade sobre grupoId — Store e tipos.
  • RN-U03: Se só divisões, clienteId = cliente do grupo da primeira divisão — resolveClienteIdFromDivisoes, getGrupoByDivisaoId.
  • RN-U04: Se só grupos, clienteId = cliente do primeiro contrato do primeiro grupo — resolveClienteIdFromGrupos.
  • RN-U05: Permissões como Record<string, boolean>; podem vir de perfil e ser persistidas por usuário — Store e API.
  • RN-U06: perfilId opcional; quando informado, persistido e exibido na listagem e edição — Store, API, page.tsx e [id]/page.
  • RN-U07: criadoEm, atualizadoEm, ultimoAcessoEm controlados pelo sistema (ISO) — Store; ultimoAcessoEm via API se houver.
  • RN-G01 a RN-G05: Estrutura do grupo (nome, responsavelId, contratoIds, divisoes); divisão (id, nome, ativoIds); múltiplos contratos e união de ativos; listagem com contagens; Cliente/Contrato como referência com opções (razão social, contrato, CNPJ) — Store, tipos, API grupos e ativos, getGruposList, DropdownClientesMultiplo.
  • RN-P01 a RN-P04: Perfil (id, nome, permissoes); aplicar perfil na matriz sem sobrescrever na primeira carga na edição; catálogo MODULOS_PORTAL; matriz somente leitura fora de Criar perfil — tipos, store, [id]/page (skipNextPerfilApply), MatrizPermissoes readOnly.
  • RN-A01 a RN-A03: Ativo com placa opcional e numeroSerie; placa sem hífen; movimentação em lote apenas dos selecionados, com persistência — tipos, store, ModalNovoGrupo moverAtivosEmLoteParaDivisao e onSalvar.
